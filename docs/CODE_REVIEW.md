### Critical

#### C-1 — Prisma schema-generated client drift

**File**: `prisma/schema.prisma` (319 lines, 15 models) vs `prisma/generated/models/EmailOutbox.ts` + `prisma/generated/models/Verification.ts`.

**Evidence**:
- `Grep "^model\\s+\\w+" prisma/schema.prisma` returns exactly 15 models — `class_schedule`, `gradelevel`, `room`, `subject`, `program`, `program_subject`, `teacher`, `timeslot`, `break_group`, `break_group_grade`, `teachers_responsibility`, `table_config`, `User`, `Account`, `Session`. No `EmailOutbox`. No `Verification`.
- `prisma/generated/models/EmailOutbox.ts` and `prisma/generated/models/User.ts` and other generated files reference `EmailOutbox`.
- `prisma/migrations/20251218091051_add_email_outbox/migration.sql:8` — `CREATE TABLE "EmailOutbox"`.
- `prisma/migrations/20260102180802_add_password_reset_email_kind/migration.sql` exists.
- `src/lib/auth.ts:116, 152, 173, 218, 254, 271` calls `prisma.emailOutbox.create / update`.

**Risk**: Build succeeds today because `prisma/generated/` is committed and contains the missing models. The first `prisma generate` against the *current* schema (which happens on a fresh CI run or any contributor who runs `pnpm db:migrate`/`prisma generate`) will produce a smaller client → TypeScript compile errors at every `prisma.emailOutbox.*` site → CI red.

**Fix**: Restore the `EmailOutbox` and `Verification` (better-auth) models in `schema.prisma`. Re-run `prisma generate`. Verify `pnpm typecheck` and `pnpm build` succeed clean. See **REFACTOR_PLAN.md Phase A.1**.

#### ~~C-2 — Two mutating actions lack auth check~~ — **WITHDRAWN (recheck 2026-05-26)**

**File**: `src/features/schedule-arrangement/application/actions/conflict-resolution.actions.ts:34-111` (`suggestResolutionAction`) and `:119-274` (`applySwapAction`).

**Correction**: Both are defined as `createAction(schema, handler)` with **no `options` argument**. `createAction` (`src/shared/lib/action-wrapper.ts:99-127`) runs `auth.api.getSession` and rejects unauthenticated callers with `UNAUTHORIZED`, then checks `allowedRoles ?? ["admin"]` — so both actions are **admin-only**. The original finding scanned the action body for an inline `getSession` and missed the wrapper. **Not a vulnerability.**

#### ~~C-3 — `updateConfigStatusAction` lacks admin role check~~ — **WITHDRAWN (recheck 2026-05-26)**

**File**: `src/features/config/application/actions/config-lifecycle.actions.ts:24-79`.

**Correction**: `updateConfigStatusAction = createAction(UpdateConfigStatusSchema, handler)` with no `options` → `allowedRoles` defaults to `["admin"]`. The wrapper enforces the admin RBAC check (`action-wrapper.ts:117-127`) before the handler runs. **Not a vulnerability.**

### High

#### H-1 — Missing `"server-only"` markers across the codebase

**Evidence**: A repository-wide search for `import` statements pulling in the `server-only` or `client-only` packages returns zero matches. `prisma.ts`, `auth.ts`, `mailer.ts`, every `*/infrastructure/repositories/*.ts`, `cache-invalidation.ts` — none import `"server-only"`.

**Risk**: A future client component that accidentally imports `prisma` or `auth` will not surface the error at build time. Worst case: secrets in the client bundle.

**Fix**: Add `import "server-only";` to the top of `src/lib/prisma.ts`, `src/lib/auth.ts`, `src/lib/mailer.ts`, `src/lib/cache-invalidation.ts`, and every file under `src/features/*/infrastructure/repositories/`. See **REFACTOR_PLAN.md Phase A.3**.

#### H-2 — No middleware; per-layout auth gating

**Evidence**: No `src/middleware.ts`. Each protected segment (`dashboard`, `schedule`, `management`) implements auth in its own `layout.tsx`.

**Risk**: A new protected route grafted in without a wrapping layout (or with a sibling layout that forgets the check) leaks. Layouts don't always re-run on client navigation if `loading.tsx`/`error.tsx` boundaries swap inside them — under specific patterns the session check can be bypassed visually until next full reload.

**Fix**: Introduce `src/middleware.ts` (or a framework-agnostic Routing Middleware) with matcher patterns for `/dashboard/:path*`, `/schedule/:path*`, `/management/:path*`. Verify session, redirect to `/signin?next=...` on miss. Keep per-layout role checks for finer-grained authorization. See **REFACTOR_PLAN.md Phase B.4**.

#### ~~H-3 — Conflict-check actions skip schema parse~~ — **WITHDRAWN (recheck 2026-05-26)**

#### M-1 — CSP requires `'unsafe-eval'`

**File**: `next.config.mjs:51-52`.

**Evidence**: CSP `script-src` contains `'unsafe-eval'`. Inline comment cites React Compiler dev mode + MUI Emotion SSR.

#### M-2 — Raw `error.message` returned in 500 responses

8 routes; full list in the findings table above.

**Risk**: Leaks Prisma error strings, table/column names, occasionally connection details. Aids enumeration.

**Fix**: Adopt the `sanitizeErrorMessage` helper already used in `/api/schedule/auto-arrange:291` and `/api/schedule/validate-drop:272`. Log the real error server-side via `createLogger(...)`. See **REFACTOR_PLAN.md Phase A.5**.

#### M-3 — Direct Prisma call in palette RSC

`src/app/schedule/[academicYear]/[semester]/arrange/@palette/page.tsx:38` calls `prisma.teachers_responsibility.findMany` with `include: { subject, gradelevel }`. The rest of the codebase routes through `*/infrastructure/repositories/*.ts`.

**Fix**: Move query into `teaching-assignment.repository.ts` (already has the join shape) and call it from the page. See **REFACTOR_PLAN.md Phase B.2**.

#### M-4 — Zustand devtools middleware unconditional

`arrangement-ui.store.ts` wraps with `devtools(immer(...))` regardless of NODE_ENV. Devtools middleware ships to prod and increases bundle size.

**Fix**: Conditional: `process.env.NODE_ENV === "production" ? immer(...) : devtools(immer(...))`. See **REFACTOR_PLAN.md Phase A.6**.

#### M-5 — Missing `revalidatePath` after mutation

`arrange.actions.ts:201` `syncTeacherScheduleAction` calls `invalidatePublicCache` (Accelerate-side) but not `revalidatePath` / `revalidateTag` (Next.js side). Public RSC routes will see stale data until the next deploy or manual refresh.

**Fix**: Add `revalidatePath('/schedule/[year]/[sem]/arrange')` (and any public viewer routes). See **REFACTOR_PLAN.md Phase A.4**.

#### M-6 — Raw DB row returned to client

`schedule-arrangement.actions.ts:163-168` returns `{ classId, created }` directly from the Prisma insert result. Currently the shape is small — but the pattern invites over-exposure as the model grows. Map to a typed DTO.

**Fix**: Define `ArrangeScheduleResult` in `presentation/types/` and return that. See **REFACTOR_PLAN.md Phase B.5**.

#### M-7 — Direct `prisma.delete` in the action layer — **DOWNGRADED to Low (recheck 2026-05-26)**

**Correction**: `unassignTeacherAction` (`teaching-assignment.actions.ts:118-134`) IS wrapped in `createAction` (auth + admin RBAC + validation), and it DOES call `revalidatePath("/management/teacher-assignment")` and `invalidatePublicCache(["teachers","stats"])`. The original "outside the wrapper / revalidation missing" claims are false. Residual issue is a layering nit only: the handler calls `prisma.teachers_responsibility.delete(...)` directly instead of going through a repository, and carries a stale `// Note: This would need additional fields…` comment.

**Fix**: Move the delete into `teaching-assignment.repository.ts`; drop the stale comment. Low priority. See **REFACTOR_PLAN.md Phase B.5**.

#### M-8 — Multi-write loop without transaction (+ false atomicity comment) — **re-verified valid**

`lock.actions.ts:144-187` `createBulkLocksAction` iterates `input.locks` calling `lockRepository.createLock(...)` one-by-one. Each call is its own implicit transaction, so a failure on lock _n_ leaves locks _0..n-1_ committed — a half-applied bulk operation. Aggravating: the loop is preceded by the comment `// Use transaction for atomicity`, which is **false** — there is no `$transaction` / `withPrismaTransaction` wrapping the loop. A future reader trusts a guarantee the code doesn't provide.

**Fix**: Wrap the loop in `withPrismaTransaction(async (tx) => { … })` so the comment becomes true, or delete the comment if per-lock commit is intentional. See **REFACTOR_PLAN.md Phase B.5**.

#### M-9 — `noUnusedLocals` / `noUnusedParameters` disabled

`tsconfig.json:47-50` documents the deferral with 73 pre-existing violations. Until re-enabled, dead variables accrete.

**Fix**: Schedule a single cleanup PR; re-enable; gate further introductions in CI. See **REFACTOR_PLAN.md Phase B.6**.

#### M-10 — 92 `as any` / `as unknown as` casts

26 files contain these. Most are clustered in repositories and a few presentation hooks. Document and budget down.

**Fix**: Track in a follow-up issue; reduce per quarter. See **REFACTOR_PLAN.md Phase B.6**.

#### M-11 — Mailer wired into `auth.ts` despite "Email disabled" policy

CLAUDE.md mandates `Email disabled: skip src/lib/mailer.ts, auth mail wiring, /management/email-outbox until mail service chosen`. Reality: `src/lib/auth.ts:19` imports `sendEmail`; lines `:143` and `:245` call it during password-reset and email-verify flows. The outbox creation succeeds; the actual send returns `SKIPPED` when ACS is not configured — but the runtime path is wired.

**Fix**: Either re-confirm the policy (and remove the mailer import/calls), or pick the mailer and remove the policy. The current state is a documentation/reality mismatch. See **REFACTOR_PLAN.md Phase C.4**.
